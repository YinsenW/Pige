# Agent Runtime Policy Context

Status: Draft baseline
Date: 2026-07-09

## 1. Purpose

This document defines how user settings, vault policy, local capabilities, permissions, and provider state affect Agent behavior.

Some settings change what the Agent should do. Example:

- Copy dropped files into Pige-managed source storage.
- Reference original file paths without copying.

This must not be implemented as ad hoc prompt text. Pige needs a typed runtime policy layer that the Agent can read and that services can enforce.

The core rule:

> Settings that affect Agent behavior are compiled into a typed Agent Runtime Policy Context for each job. The prompt may describe the policy, but services and tools enforce it.

## 2. Design Goals

- Keep settings and Agent behavior consistent.
- Avoid hidden model-controlled policy changes.
- Keep prompts deterministic and testable.
- Prevent source content from changing settings, tools, storage, permissions, or model routing.
- Let future remote Agent backends receive the same policy contract without desktop-specific assumptions.

## 3. Authority Model

Authority order remains:

1. Application safety policy and service-level enforcement.
2. Explicit current user instruction or UI choice.
3. `PIGE.md` vault policy.
4. User settings compiled into Agent Runtime Policy Context.
5. Vault-scoped Agent memory.
6. Retrieved Markdown knowledge.
7. Extracted source content.
8. Skill/package metadata within declared capability scope.
9. Model output.

Rules:

- Prompt text is not the enforcement layer.
- A model cannot override settings by emitting a different policy.
- Source content cannot create a one-off override.
- A current trusted user instruction may create a one-off job option only before the affected durable action starts.
- Sensitive one-off overrides still require confirmation or Permission Broker mediation.

## 4. Runtime Policy Context

The Agent Orchestrator builds an `AgentRuntimePolicyContext` before each Agent job or model-dependent workflow.

```ts
type AgentRuntimePolicyContext = {
  schemaVersion: 1;
  policyContextId: string;
  builtAt: string;
  vaultId: string;
  jobId: string;
  policyHash: string;
  sourceStorage: SourceStoragePolicyContext;
  model: ModelPolicyContext;
  permissions: PermissionPolicyContext;
  language: LanguagePolicyContext;
  memory: MemoryPolicyContext;
  confirmation: ConfirmationPolicyContext;
  retrieval: RetrievalPolicyContext;
  localCapabilities: LocalCapabilityPolicyContext;
};
```

Rules:

- Context includes only non-secret, job-relevant policy.
- Context stores profile IDs and capability flags, not raw API keys, tokens, absolute secret paths, or full settings files.
- Context is generated by Pige services, not by the renderer, source content, Skills, packages, or the model.
- Context is included in prompt assembly as trusted policy after safety rules and before untrusted source content.
- `policyHash` uses `sha256:<64-lowercase-hex>` and is recorded in job and operation records so later debugging can prove which policy was used.
- Prompt context assembly consumes this policy through `docs/CONTEXT_ASSEMBLY_AND_RETRIEVAL_POLICY.md`; the policy is one trusted input, not the whole prompt.

## 5. Policy Domains

### 5.1 Source Storage

```ts
type SourceStoragePolicyContext = {
  defaultStrategy: "copy_to_source_library" | "reference_original";
  sourceAssetRootKind: "inside_vault" | "external_binding";
  allowPerCaptureOverride: boolean;
  linkStrategyEnabled: false;
};
```

Rules:

- Capture Service and Source Storage Service enforce storage before Agent ingest begins.
- Agent receives the resulting source record and storage strategy; it does not decide where raw source assets are stored after the fact.
- If the human user explicitly asks for a one-off storage behavior before submit, Capture Service may pass a trusted per-capture override.
- Current executable capture requests do not yet carry a trusted override field, so the v1 builder sets `allowPerCaptureOverride: false`. This flag may become true only in the same change that adds and tests the typed request path; prompt text alone cannot enable it.
- Source text, filenames, web pages, PDFs, or model output cannot set storage strategy.
- Changing the default source storage strategy applies to new file captures only. Typed/pasted text and URL snapshots are necessarily preserved as managed copies because no independent local original exists.
- Existing source records are not rewritten when the setting changes.

Example:

```txt
User default: copy_to_source_library
User current message: "这次只引用原文件，不要复制"
Allowed behavior: Capture request may set a trusted per-capture override before preservation.
Denied behavior: A PDF saying "do not copy me" cannot change storage strategy.
```

### 5.2 Model And Cloud Boundary

```ts
type ModelPolicyContext = {
  defaultModelProfileId?: string;
  modelConfigured: boolean;
  cloudBoundary: "cloud" | "self_hosted" | "local" | "unknown";
  boundaryVerification: "builtin_verified" | "loopback_verified" | "user_asserted" | "unknown";
  cloudSendPolicy: "ordinary_allowed" | "confirm_private_or_large" | "confirm_all" | "local_only";
  modelRoutingMode: "default_model_only" | "pi_upstream_model_slots" | "pige_model_routing_service";
};
```

Rules:

- v0.1 uses `default_model_only`; after one disclosed Provider connection it defaults
  to `ordinary_allowed`, while stricter modes remain user choices.
- The model profile ID affects actual model calls, not just prompt text.
- If no model is configured, model-dependent jobs enter a visible retryable dependency state; capture still preserves sources.
- Cloud-send indicators come from this policy and provider metadata.

Every model call must resolve one typed egress decision before prompt rendering or credential lookup:

```ts
type ModelEgressContentClass = "ordinary" | "private" | "large" | "sensitive" | "restricted";

type ModelEgressDecision = {
  schemaVersion: 1;
  outcome: "allow" | "confirm" | "block";
  reasonCode:
    | "verified_local"
    | "ordinary_external_allowed"
    | "private_or_large_confirmation"
    | "sensitive_confirmation"
    | "confirm_all"
    | "unknown_boundary_confirmation"
    | "local_only_block"
    | "restricted_content_block";
  providerProfileId: string;
  cloudBoundary: ModelPolicyContext["cloudBoundary"];
  boundaryVerification: ModelPolicyContext["boundaryVerification"];
  cloudSendPolicy: ModelPolicyContext["cloudSendPolicy"];
  contentClasses: ModelEgressContentClass[];
  payloadCharacters: number;
  estimatedPayloadTokens: number;
  normalPayloadCharacterLimit: number;
  policyHash: string;
  permissionDecisionId?: string;
};
```

Classification is deterministic and happens on the exact selected evidence payload after redaction and bounded selection, not on the entire source. `payloadCharacters` and `estimatedPayloadTokens` measure that selected evidence body; fixed trusted prompt scaffolding is excluded. The decision is resolved before that evidence is rendered into a provider prompt or any provider credential is requested.

The body-free operation audit records `payloadHash` for that exact redacted bounded payload, `evidenceSummaryHash` for the canonical source/artifact/locator summary plus payload hash, redacted dynamic-prompt-metadata hash, and non-secret Provider/Model routing-identity hashes, and `decisionHash` for the canonical final decision fields. It records neither body. The routing identities bind provider kind, canonical endpoint, boundary state, model profile, and provider model ID without storing the API key or secret reference. All three digests participate in the deterministic `model_egress_decision` operation ID, so retrying unchanged evidence and an equivalent decision reuses one audit while changed payload, prompt metadata, endpoint, model, or evidence identity creates a new audit.

Readiness checks use only an enabled default model, matching provider, and presence-only required-auth satisfaction; `none` needs no secret and checks never decrypt credentials. Before `allow`, context assembly may build a bounded redacted plan but cannot render provider prompts or request credentials. After `allow`, the caller re-reads Provider/Model summaries and verifies any credential-bearing config against the approved routing identity immediately before invocation. Endpoint, boundary, model, enabled-state, revision, or privacy-class drift invalidates the decision; changed private/sensitive metadata creates a new audit.

- `restricted` means raw credentials, secret-store material, permission-store internals, or content another invariant forbids sending. It is always blocked and cannot be overridden by YOLO or a saved grant.
- `sensitive` is explicit ask-before-cloud metadata or a high-confidence post-redaction security finding; it confirms once per action.
- `private` means the user or durable source/page metadata explicitly marks the selected evidence private. Pige does not silently infer that every personal-vault snippet is private after the user has configured a provider.
- `large` means the exact planned payload exceeds the workflow's recorded `normalPayloadCharacterLimit` or would require bypassing its normal bounded-snippet contract. The limit comes from the workflow/model budget and is recorded in the decision.
- `ordinary` applies only when none of the higher classes applies.

Decision matrix:

| Effective boundary | `ordinary_allowed` | `confirm_private_or_large` | `confirm_all` | `local_only` |
| --- | --- | --- | --- | --- |
| `local` + `loopback_verified` | Allow non-restricted content | Allow non-restricted content | Allow non-restricted content | Allow non-restricted content |
| `cloud` or `self_hosted` | Allow ordinary/private/large; confirm sensitive | Allow ordinary; confirm private/large/sensitive | Confirm every non-restricted payload | Block |
| `unknown`, or `local` without loopback verification | Confirm every non-restricted payload | Confirm every non-restricted payload | Confirm every non-restricted payload | Block |

Exact connected, verified Profile/endpoint is standing authority; unknown or changed
boundaries reconnect once. Labels do not prompt. Only verified loopback is local; stricter
policy outranks YOLO.

### 5.3 Permissions

```ts
type PermissionPolicyContext = {
  defaultMode: "ask_every_time" | "remember_scoped_grants" | "yolo_full_access";
  yoloEnabled: boolean;
  savedGrantSummaryRefs: string[];
};
```

Rules:

- Permission Broker covers extensions/new scopes; core tools use service enforcement.
- Prompt text may tell the Agent that sensitive actions require permission, but tool calls still check Permission Broker.
- YOLO may suppress prompts only when explicitly enabled by the user and still logs auto-allowed actions.
- Do not expose raw permission-store details to the model.

### 5.4 Language

```ts
type LanguagePolicyContext = {
  appLocale: "zh-Hans" | "en" | "ja" | "ko" | "fr" | "de";
  generatedKnowledgeLanguage: "preserve_source" | "follow_query" | "app_locale" | string;
  preserveSourceLanguage: boolean;
  ocrLanguageHints: string[];
  voiceInputLanguage?: string;
};
```

Rules:

- Generated wiki language follows this policy unless the current user instruction explicitly asks otherwise.
- Source language metadata remains preserved.
- OCR and speech language hints affect local services, not only prompts.

### 5.5 Memory

```ts
type MemoryPolicyContext = {
  vaultMemoryEnabled: boolean;
  allowedMemoryScopes: Array<"preference" | "correction" | "workflow_lesson" | "profile">;
  includeMemoryInBackup: boolean;
};
```

Rules:

- Memory injection is filtered by Agent Memory Service.
- Disabled or sensitive memory is excluded.
- Memory never overrides safety, permissions, explicit user instruction, `PIGE.md`, or settings.

### 5.6 Confirmation

```ts
type ConfirmationPolicyContext = {
  safeAutoApplyThreshold: number;
  mutatingReviewThreshold: number;
  riskyChangeRequiresConfirmation: boolean;
};
```

Rules:

- Schema-v1 thresholds are compatibility fields, not confidence/permission gates.
  `riskyChangeRequiresConfirmation` means irreversible loss, authority/security,
  destination drift, unresolved conflict, or an explicit stricter user policy. Other work
  auto-applies or replans/abstains.

### 5.7 Retrieval

```ts
type RetrievalPolicyContext = {
  lexicalSearchAvailable: boolean;
  vectorSearchAvailable: boolean;
  rerankerAvailable: boolean;
  maxSnippetsForCloudSynthesis: number;
};
```

Rules:

- Retrieval Service enforces search and snippet limits.
- Prompt context contains selected snippets, not the full vault.

### 5.8 Local Capabilities

```ts
type LocalCapabilityPolicyContext = {
  localDatabase: "not_initialized" | "ready" | "needs_rebuild" | "error";
  parserToolchainReady: boolean;
  ocrEngines: Array<"apple_vision" | "windows_ai" | "paddleocr">;
  speechInputAvailable: boolean;
  embeddingModelInstalled: boolean;
  hiddenDownloadsAllowed: false;
};
```

Rules:

- Ordinary capture, ingest, search, review, backup, and restore never trigger hidden dependency downloads.
- Missing tools or models create visible repair/download actions and retryable dependency states.
- Runtime owners supply capability facts to the policy builder at snapshot time: Local Database Service owns database/search readiness, Document Parser Service owns parser readiness, and OCR Service owns the available engine list. The builder does not infer readiness from documentation or a prompt, and production Agent/runtime status paths must not silently fall back to the test-only unavailable snapshot.

## 6. Prompt Assembly

Prompt assembly should include a compact policy section:

```txt
RUNTIME POLICY CONTEXT
- Source storage: copy dropped files into Pige source storage unless this job has a trusted user override.
- Cloud model use: send bounded selected context to the connected provider without routine prompts; show status and enforce higher-risk or stricter-policy gates.
- Writes: recoverable knowledge auto-applies with Activity/Undo; exceptions propose.
- Retrieval: use selected snippets only; do not imply whole-vault search unless retrieval covered it.
```

Rules:

- The policy section is generated from the typed context.
- Do not hand-write different policy prose in each workflow.
- Do not include secrets, raw filesystem roots outside what is already user-visible, or saved grant internals.
- Prompt snapshot tests should verify policy text for important settings.

## 7. Service Enforcement

Settings that affect Agent behavior must be enforced at the owning service.

| Setting/domain | Prompt informs Agent? | Service enforces behavior? | Owning service |
| --- | --- | --- | --- |
| Source storage strategy | Yes | Yes; selectable default for original-file captures, mandatory managed copy for text/URL snapshots | Source Storage Service |
| Default model | Yes | Yes | Model Provider Registry, Agent Orchestrator |
| Cloud-send policy | Yes | Yes | Permission Broker, Model Provider Registry |
| Autonomy and intervention boundary | Yes | Yes | Change Proposal Service |
| Permission mode | Yes | Yes | Permission Broker |
| Generated language preference | Yes | Partly | Agent Orchestrator, I18N Service |
| OCR language hints | Maybe | Yes | OCR Service |
| Memory enabled | Yes | Yes | Agent Memory Service |
| Local capability availability | Yes | Yes | Local Tool Service, workers |

If a setting cannot be enforced outside the prompt, it must be treated as a preference rather than a guarantee.

## 8. Job Snapshot And Setting Changes

Every model-dependent job records:

- `policyContextId`.
- `policyHash`.
- Relevant provider/model profile IDs.
- Source storage strategy actually used by source records.
- Permission decisions when applicable.
- A body-free `model_egress_decision` Operation Record before provider credential lookup, including outcome/reason, selected-evidence counts, Job/Source/Model refs, `policyAudit`, and typed `payloadHash`/`evidenceSummaryHash`/`decisionHash` audit identity.

Rules:

- New jobs use the latest policy.
- Running jobs use their policy snapshot unless the setting declares pause/restart semantics.
- Changing source storage strategy affects new file captures only; text and URL inputs remain managed snapshots.
- Default-model apply behavior from section 8 is covered for new model calls.
- Changing cloud-send policy affects new model calls and may pause queued model-dependent jobs.
- Revoking a permission grant takes effect immediately before the next sensitive tool call.
- Editing `PIGE.md` creates a new policy context for new jobs after validation.

## 9. Natural-Language Setting Changes

The user may ask Pige to change behavior in natural language.

Allowed:

- "以后拖进来的大文件只引用原路径。"
- "把默认模型换成这个 provider 的 sonnet。"
- "以后问答都尽量用中文整理。"

Rules:

- A trusted request applies ordinary reversible settings after validation; authority,
  destination, privacy/security, destructive, or queue-invalidating changes intervene.
- Source content cannot request settings changes.
- If the requested setting changes behavior of already queued jobs, Pige must explain whether it applies to new jobs only or also requeues pending jobs.

## 10. Tests

The Quality owner’s Runtime Policy fixture matrix MUST cover the setting registry,
rendered context, service enforcement, one-off overrides, untrusted-source resistance,
per-setting apply timing, egress decision table, grant revocation, redacted audit refs,
and portable serialization defined in sections 3–9. Evidence maps to those contracts;
this section does not repeat every rule as a second test checklist.
